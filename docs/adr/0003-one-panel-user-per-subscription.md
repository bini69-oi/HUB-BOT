# 0003 — Один panel-user на подписку

**Статус:** Accepted

## Контекст

Remnawave энфорсит HWID/device-лимит на уровне panel-user'а. Если у пользователя несколько тарифов
(мульти-тариф) и все маппятся на одного panel-user'а — они делят наименьший device-лимит и портят
друг друга. Проверенное решение — отдельный panel-user на каждую подписку.

## Решение

Каждая `Subscription` = свой **panel-user**. У подписки есть **постоянный** уникальный суффикс
`short_id` (VARCHAR(16), UNIQUE), которым шаблонятся `username`/`description` panel-user'а.
`short_id` **НЕ** выводится из мутабельного `id` (никаких `f"sub{id}"`). Активная подписка на план
энфорсится partial-unique индексом: `WHERE status IN (active, trial, limited)`.

## Последствия

- (+) Мульти-тариф не ломает HWID-лимиты.
- (+) Ротация sub-URL/revoke бьёт по нужному panel-user'у.
- (−) Больше panel-user'ов, чем клиентов; синк/reconcile обязателен (см. ADR-0005).
- `short_id` генерируется при создании и **не меняется** весь жизненный цикл подписки.
