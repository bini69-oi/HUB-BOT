## [1.6.9] — 2026-07-21

Исправления по аудиту (деньги/формы).

### Исправлено

- **Двойное списание сохранённой карты в автоплатеже.** Пометка «попытка списания сегодня»
  ставилась вне блокировки строки, поэтому два пересекающихся запуска автоплатежа оба проходили
  проверку «ещё не списывали» и списывали карту дважды за одно продление. Пометка теперь ставится
  и коммитится ВНУТРИ блокировки — параллельный запуск её видит и не списывает повторно.
- **Случайный текст мог оформить вывод средств.** Если пользователь начал вывод (бот ждёт
  реквизиты), а затем нажал кнопку меню/покупки (которая не сбрасывала форму), следующее
  сообщение оформлялось как реквизиты вывода с этим текстом. Теперь нажатие любой кнопки
  навигации/покупки сбрасывает незавершённую форму — как это уже работало в админ-панели.

## [1.6.8] — 2026-07-21

Фокусный аудит: обновление + настройки. Закалка механизма самообновления и границы настроек.

### Исправлено

- **Самообновление отключало само себя после первого использования (КРИТИЧНО).** После
  обновления сайдкар делал `chown -R` по всей папке репо и задевал вложенный том сигналов,
  делая его root-овым — после чего непривилегированный процесс бота больше не мог положить
  маркер, и кнопка «Обновить»/автообновление молча переставали работать. Том сигналов теперь
  исключён из смены владельца.
- **Кнопка «Обновить» больше не врёт «запущено», когда модуль updater выключен.** Готовность
  определялась по наличию папки (она есть всегда), а не по живости сайдкара — теперь сайдкар
  шлёт heartbeat, и при его отсутствии честно сообщается «updater не подключён».
- **Ручное и авто-обновление больше не конфликтуют.** Блокировка (`flock`) стояла в `/tmp`,
  который не общий между хостом и сайдкаром; перенесена в репо (общий bind-mount).
- **Проверка подписи коммита не ломает авто-обновление.** В образ сайдкара добавлен `gnupg`
  (без него `git verify-commit` падал даже на корректно подписанных коммитах при
  `UPDATE_VERIFY_SIGNATURE=1`).
- **На systemd-установке пропадает вечное «доступно обновление».** `deploy.sh` теперь
  прописывает `APP__BUILD_SHA` в `.env`, чтобы версия определялась корректно.
- **Реферальный процент ограничен 0–100.** Опечатка вроде `150` больше не начисляет рефереру
  больше, чем заплатил пользователь.
- **`BACKUP_KEEP_LAST` не может быть 0** (иначе удалялись все копии, включая свежую).

### Прочее

- Автоуведомление об обновлении при выключенном updater больше не повторяется каждые 6 ч.
- Категория «ИИ-поддержка» в настройках получила иконку и подпись; кнопка «Перезапуск панели»
  переименована в «Перезапуск веб-кабинета» (она рестартит веб-кабинет, а не панель Remnawave).

## [1.6.7] — 2026-07-21

Механизм обновления и обслуживания — чтобы кнопки реально работали, а не показывали «✓» впустую.

### Исправлено

- **Кнопки в web-кабинете «Обслуживание» ничего не делали.** «Обновить бота», «Перезапустить»
  писали зелёную галочку, но не выполняли действие (в отличие от кнопки «Обновить» в самом
  боте). Теперь они действительно запускают действие через модуль обновлений (тот же маркер,
  что у кнопки в боте): «Обновить» — pull+пересборка+рестарт; «Перезапустить бота/панель» —
  перезапуск сервиса без пересборки. Статус честный: «запущено», «модуль updater не подключён»
  (с командой для ручного запуска) или «вручную» для перезагрузки сервера (её из контейнера
  безопасно не сделать).

### Добавлено

- **Перезапуск сервисов через модуль обновлений.** `updater` теперь понимает не только
  обновление, но и `restart <сервис>` (bot/web/worker/scheduler/all) — `docker compose restart`
  без пересборки, для быстрого перезапуска из кабинета.

## [1.6.6] — 2026-07-21

Продолжение исправлений по аудиту (средние/низкие).

### Исправлено

- **Полный бэкап БД больше не уходит в Telegram под слабым ключом.** Архив со всеми данными
  (почты, хэши паролей, токены карт, балансы) отправлялся в отчётную группу, зашифрованный
  запасным ключом из JWT-секрета. Теперь без выделенного `BACKUP_ENCRYPTION_PASSWORD` бэкап
  делается только локально и НЕ отправляется вовне (с ключом — отправляется как прежде).
- **Пароль БД больше не виден в списке процессов.** `pg_dump` получал пароль в аргументах
  командной строки (виден через `ps`/`/proc`); теперь — через переменную окружения `PGPASSWORD`.
- **Двойное списание с баланса при медленной панели.** Защита от двойного нажатия жила 15 сек,
  а провижининг с ретраями панели мог длиться до ~60 сек — второе нажатие успевало войти и
  списать повторно. TTL защиты поднят до 90 сек.
- **Смена ссылки подписки не блокируется на 10 минут при сбое панели.** Раньше лимит ставился
  до вызова панели и при её сбое юзер не мог повторить 10 минут; теперь лимит снимается при сбое.
- **`/setbanner` с опечаткой в названии экрана** больше не перезаписывает молча общий баннер —
  неизвестное имя отклоняется со списком доступных экранов.
- **Отрицательный лимит активаций промокода** через PATCH больше не принимается (как и при создании).

## [1.6.5] — 2026-07-21

Полный аудит бота (6 направлений). Ниже — критичные и высокоприоритетные исправления;
остальные (средние/низкие) — следующими патчами.

### Исправлено

- **Смена тарифа: пере-начисление и потеря дней (переработан пересчёт).** Прежний расчёт
  оценивал остаток по ставке последнего платежа, из-за чего годовой тариф с коротким
  доп-платежом при смене раздувал остаток (сотни бесплатных дней = утечка дохода); а при
  «промахе» поиска (окно последних транзакций) оплаченные дни, наоборот, молча пропадали.
  Теперь пересчёт детерминированный: остаток берётся из срока самой подписки и конвертируется
  по КАТАЛОЖНЫМ ставкам тарифов (текущий — по самой дешёвой ставке, чтобы не пере-начислять;
  новый — по цене выбранного периода). Если цена неизвестна — остаток сохраняется как есть
  (ведёт себя как продление). Ни пере-начисления, ни потери дней.
- **Кнопка «Личный кабинет» роняла бота в режиме нижней панели.** Диспетчер вызывал обработчик
  кабинета без обязательного аргумента (FSM-состояние) — каждое нажатие давало ошибку, а сам
  кабинет был недостижим в reply-режиме. Исправлено; добавлен тест совместимости сигнатур.
  Заодно неизвестный код действия в нижней панели больше не «залипает», а возвращает в меню.
- **Вход в админ-кабинет теперь под rate-limit.** Ранее эндпоинт входа не имел никакого
  ограничения — неограниченный перебор пароля к учётке, управляющей деньгами, данными и
  самообновлением. Добавлены окна по IP и по логину (как у входа в клиентский кабинет).
- **Обход лимитов через подделку `X-Forwarded-For`.** Идентификация клиента брала ЛЕВЫЙ хоп
  заголовка, который клиент может подделать (nginx/Caddy его дописывают, а не заменяют), —
  ротацией фейкового заголовка сбивались все лимиты (перебор/регистрация/enumeration). Теперь
  берётся `X-Real-IP` либо правый хоп (значение, добавленное нашим прокси).

## [1.6.4] — 2026-07-20

### Исправлено

- **Загрузка медиа в кабинете падала с ошибкой прав (Docker).** Папка `uploads` монтируется
  как том Docker; если её не было в образе на момент назначения владельца, том создавался от
  root, и процесс приложения (непривилегированный) не мог в него писать — загрузка баннера/
  обложки падала. Теперь `uploads`, `backups` и `update-signals` создаются в образе с нужным
  владельцем (свежие тома наследуют его), а сама загрузка при отказе прав отвечает понятным
  сообщением с командой починки вместо «внутренней ошибки». Для уже существующих установок:
  `docker compose run --rm --user root web chown -R app:app /app/uploads /app/backups`.
- **Транзиентный таймаут сети Telegram больше не летит в телеметрию как краш.** Разовый
  `TelegramNetworkError` (таймаут запроса к Telegram) — это транспортный сбой, а не баг;
  теперь он обрабатывается штатно (как flood-wait и блокировка бота) — без кода ошибки
  пользователю и без записи в телеметрию.

## [1.6.2] — 2026-07-18

### Исправлено

- **Смена тарифа сбрасывала срок подписки и странно считала цену.** При переходе на другой
  тариф подписка сбрасывалась на срок нового тарифа (например, с 60 оставшихся дней до 30),
  а остаток старого периода превращался в скидку на цену — из-за чего цена «плавала»
  (например, продление на месяц при 10 оставшихся днях считалось как за 20 и давало 30 дней
  вместо 40). Теперь смена тарифа работает честным пересчётом: оплачивается полная цена
  выбранного периода нового тарифа, а неиспользованная стоимость старого периода
  конвертируется в **бонусные дни** нового тарифа по его цене и **прибавляется** к сроку —
  дни больше не теряются. Пример: 60 дней тарифа A → переход на более дорогой B (30 дней) =
  30 купленных + бонус от остатка A. Продление тем же тарифом, как и раньше, просто прибавляет
  срок к остатку. Заодно закрыта ошибка, при которой остаток можно было переиспользовать для
  почти бесплатных продлений.

## [1.6.1] — 2026-07-18

Инцидент «оплатил — подписка появилась через ~10 минут»: ускорено восстановление
застрявших платежей (~2–3 мин в худшем случае) + полный проход по платёжному UX бота —
ни одного молчаливого исхода: каждый путь оплаты теперь говорит с покупателем.

### Исправлено

- **Застрявший платёж зачисляется за ~2–3 мин вместо 3–13.** Любой сбой быстрого пути
  (вебхук → очередь → воркер) ронял транзакцию в PENDING, и её добирал только 5-минутный
  реконсилер с 3-минутным порогом. Теперь реконсилер тикает каждую минуту (свежее
  30-минутное окно; полный суточный свип ~каждый 5-й тик по персистентному маркеру —
  пропущенный тик откладывает хвост, а не теряет его), порог «застрял» `3 мин → 90 с`,
  лок с токеном владения против наложения свипов, курсор пагинации — брошенные инвойсы
  больше не занимают окно бесконечно.
- **Ретраи выдачи переживают блип панели.** Транзиентная ошибка ретраится внутри задачи
  с паузами 5/15 с (раньше 5 мгновенных ретраёв брокера сгорали за секунды и не
  переживали даже 30-секундный сбой); воркеру дан `stop_grace_period: 120s` — рестарт
  или автообновление больше не убивает платёж посреди ретрая (сообщение брокера
  терялось навсегда: ack-on-pickup).
- **Вебхук-роут не роняет проверенный платёж на блипе Redis.** Постановка в очередь
  ретраится; при финальном сбое отвечаем 200 только если реконсилер реально восстановит
  счёт (поллящийся шлюз + счёт моложе 23 ч), иначе 503 — провайдер передоставит.
- **YooKassa: сбой верификационного рефетча (сеть/5xx) ретраится инлайн** вместо
  мгновенного 403, который парковал реальный платёж до реконсилера.
- **Stars: повторная оплата старого инвойса.** `pre_checkout` проверяет счёт ДО списания
  звёзд и отклоняет отсутствующие/уже обработанные; повторное списание отвечает честно
  (а не ложным «✅ Баланс пополнен») и алертит админов; `telegram_payment_charge_id`
  сохраняется — поддержке есть чем сделать возврат; упавшая выдача ставится в
  durable-очередь с ретраями вместо вечного PENDING.
- **Умная корзина больше не пропадает молча.** Забор корзины атомарный (два
  одновременных зачисления не оформят покупку дважды), при ошибке интент сохраняется,
  пользователь получает «не хватает ещё» / «оформи вручную», админы — алерт.
- **Платёж нельзя молча задропить.** Рейт-лимит и режим техработ пропускают
  settlement-апдейты (задропленный `successful_payment` = списанные звёзды без
  зачисления); во время техработ оплата отклоняется ДО списания; задропленный колбэк
  всегда отвечается — кнопки не крутятся вечно.
- **Дабл-тап оплаты** закрыт пер-юзер локом — двойного списания с баланса и дублей
  инвойсов больше нет.
- **`BALANCE_ENABLED` гейтит все входы** (старые кнопки/кастом-меню позволяли класть
  деньги в отключённый кошелёк); кованая сумма пополнения ре-валидируется по
  `MIN_DEPOSIT_AMOUNT`; отказ Telegram создать инвойс отменяет сиротский PENDING
  вместо вечного «⏳ Пополнение» в истории.

### Добавлено

- **Кнопка «Проверить оплату» на экране счёта** — он-деманд опрос провайдера через тот
  же идемпотентный пайплайн, что у реконсилера: потерянный вебхук = один тап вместо
  многоминутного ожидания. Кулдаун 10 с на счёт бережёт API провайдера.
- **Уведомления о негативных исходах платежа**: новые редактируемые шаблоны
  `payment_failed` / `payment_canceled` / `payment_underpaid` (недоплата дополнительно
  всегда алертит админов — деньги-то реально пришли), `auto_purchase` /
  `auto_purchase_failed` для умной корзины. Раньше отклонённый или недоплаченный платёж
  заканчивался полной тишиной.
- **Экран «не хватает средств»** показывает цену, баланс и сколько именно не хватает,
  плюс срок жизни резерва покупки; кнопка «Пополнить» есть и при выключенной автопокупке
  (раньше — тыкабельный тупик с тостом).
- **Stars-ответы рендерятся шаблонами владельца** (`balance_topup`, `purchase` /
  `renewal` / `plan_changed` / `traffic_topup`) — продление говорит «продлена», а не
  генерик «активирована»; тексты правятся из кабинета.
- **Телеметрия**: панельные блипы больше не сыплются как краши, но финальный провал
  платёжных задач репортится всегда, даже транзиентный.

## [1.6.0] — 2026-07-17

Второй заход аудита безопасности — оплата и обновления (после v1.4.0/v1.5.1).

### Безопасность

- **Недоплата на редактируемых формах.** Порог недоплаты подняли `0.90 → 0.95`. Только YooMoney
  отдаёт сумму «после комиссии» (quickpay, где плательщик может ПРАВИТЬ сумму), и его комиссия ≤3%,
  т.е. честный платёж — это ≥97%. Остальные шлюзы репортят gross (== счёт). Старые 0.90 позволяли
  занизить оплату примерно на 10% и получить полный продукт; 0.95 держит запас над комиссией
  YooMoney и вдвое режет утечку.
- **YooKassa: валидация `object.id` до аутентифицированного рефетча.** Вебхук YooKassa не подписан;
  его `object.id` подставлялся в путь исходящего GET к API с нашими ключами. Теперь id проверяется
  на UUID-форму (hex+дефис) — исключает инъекцию сегментов пути (`../`, лишние слэши).
- **Проверка суммы — единая неизбежная точка.** Сумма теперь пробрасывается и на пути реконсилятора
  (раньше — только живой вебхук), с проверкой валюты, чтобы чужую валюту не сравнивать с рублёвыми
  копейками. Закрывает будущую дыру: реконсилируемый + редактируемый шлюз не проскочит мимо порога.
- **Fail-closed токена сохранённой карты.** Без `APP__CRYPT_KEY` сырой charge-токен больше не
  кладётся в брокер/БД (прод и так не стартует без ключа — защита для криво настроенного dev).
- **Опциональная проверка подписи обновлений.** С `UPDATE_VERIFY_SIGNATURE=1` в `.env` `update.sh`
  делает `git verify-commit HEAD` и не собирает неподписанный коммит. По умолчанию выключено, чтобы
  существующие установки продолжали обновляться; включай, когда начнёшь подписывать релизы.

### Исправлено

- **Проверка версий всегда показывала «доступно обновление».** В образ печётся 7-символьный SHA
  (`git rev-parse --short`), а с GitHub бралось 12 символов, и `!=` был истинным даже на идентичном
  коммите: кнопка «У вас последняя версия» была недостижима, а автообновление перезапускалось каждые
  6 ч без конца (бэкап+пересборка каждый цикл, диск пух). Теперь сравнение префиксное — как git
  резолвит сокращённый SHA.
- **Ротация pre-update бэкапов.** `update.sh` хранит последние 5 дампов `pre-update-*.sql.gz` —
  раньше они копились без предела и могли заполнить диск (что само по себе ломает следующее обновление).
- **SIGPIPE-обрыв обновления.** Косметическая строка `git log … | head` под `set -o pipefail` на
  большом диапазоне коммитов убивала `git log` (SIGPIPE) и валила обновление после `git pull`, но до
  пересборки. Добавлен `|| true`.

### Добавлено

- **Уведомление владельца об итоге обновления.** `update.sh` шлёт владельцам сообщение в Telegram —
  ✅ при успехе и ⚠️ при провале/прерывании (раньше провал кнопки «Обновить» был молчаливым: лог падал
  в том, который никто не смотрит). В образ сайдкара добавлен `curl`; текст кнопки больше не обещает
  успех безусловно.
- **`flock` против гонок обновления.** Запуск с хоста и сайдкар (или 6-часовая автопроверка) больше
  не могут одновременно `docker compose build/up` один проект — второй ждёт до 30 мин, потом отступает.

- **Кратковременные обрывы соединения с Redis** больше не приводят к ошибке: клиент
  повторяет операцию с экспоненциальной задержкой.
- **Диагностика синхронизации нод.** Неуспех синхронизации логировался с пустым текстом

## [1.2.1] — 2026-07-15

Изменения по обратной связи операторов: настройка экрана подключения, продление
подписки на произвольный срок и исправление двух ошибок — развёртывания и интерфейса.

### Добавлено

- **Выбор клиентских приложений на экране «Подключение».** Оператор задаёт, какие
  приложения предлагать пользователю для импорта подписки и в каком порядке — настройкой

- **Продление подписки на произвольный срок.** В карточке пользователя, помимо кнопок
  +7 / +30 / +90 дней, добавлены поле ввода количества дней и выбор конкретной даты
  окончания (календарь); дату можно как продлить, так и сократить.

### Исправлено

- **Порядок кнопок в конструкторе меню бота не сохранялся.** После нажатия «Сохранить»
  кнопки возвращались к порядку создания: редактор не передавал позицию (`order_index`),
  и сервер восстанавливал её из порядка элементов в запросе. Порядок теперь сохраняется.
- **Кнопка «Открыть в приложении» не открывала приложение.** Переход по ссылке с
  пользовательской схемой (`happ://`, `hiddify://…`) через `location.href` не срабатывает
  внутри WebView Telegram; открытие переведено на `Telegram.openLink` с запасным вариантом.
- **Установка без домена отдавала пустую страницу.** На сайте `:80` присутствовала
  директива TLS, из-за чего Caddy ожидал TLS-соединение на 80-м порту и обрывал обычные
  HTTP-запросы без ответа (`NS_ERROR_NET_EMPTY_RESPONSE`). Конфигурация исправлена: `:80`
  отдаёт HTTP, домен по-прежнему получает автоматический HTTPS.
- **Бот показывал код ошибки при нажатии на устаревшую кнопку.** Ожидаемые ответы
  Telegram API (истёкший callback-запрос, отсутствие сообщения для удаления или
  редактирования) трактовались как сбой. Теперь они обрабатываются штатно — без сообщения
  об ошибке пользователю и без записи в телеметрию.
